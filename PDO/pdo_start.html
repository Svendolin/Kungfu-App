PDO Startseite

Es würde den Rahmen meines Tutorials sprengen, wenn ich hier ein komplette Einführung in PDO (= PHP Data Objects) geben würde. Auf dieser Seite findest du nur kurze Aussagen darüber, was PDO ist und was die Vorteile beim Einsatz sind.

Die Ehe von PHP und mySQL-Datenbanken gerät in die Jahre

Die Hochzeit von PHP und mySQL war von Beginn weg eine eigentliche Erfolgsstory. Webentwickler haben mit diesem Gespann die Möglichkeit, eine extrem leistungsfähige Datenbank einzusetzen und gleichzeitig den Entwicklungsaufwand klein zu halten. Queries - Datenbankabfragen - lassen sich leicht formulieren. Leider lassen sich viele Web-Applikationen auch leicht knacken, weil viele Entwickler eben zu leichtfertig mit dem Code umgehen. Mit sog. "SQL injections" kann eine Datenbank dazu gebracht werden, Ihre Geheimnisse einem Hacker zu verraten.

Datenbankwerkzeuge in PHP

Die Sicherheit zu erhöhen ist darum ein erklärtes Ziel von den Leuten, die PHP entwickeln. Fast zur gleichen Zeit wurden deswegen zwei neue Möglichkeiten eingeführt, wie PHP mit Datenbanken "reden" kann: mysqli und PDO. Die mysqli-Schnittstelle ist eigentlich "nur" eine Verbesserung der veralteten mysql-Funktionen, während dem PDO noch mehr kann: Mit Hilfe von PDO lassen sich sogar auch andere Datenbanken als mySQL ansprechen, solange dafür ein Treiber zur Verfügung steht.

Höhere Sicherheit dank "prepared statements"

Prepared statements machen Datenbank-Abfragen sicherer. Im Prinzip wird eine Query aufgetrennt in eine Rohform (oder Schablone) mit Platzhaltern und die Parameter, die zur Abfrage gehören. Die Parameter werden vorerst überprüft, ob sie unerlaubte Dinge enthalten und erst darauf weiter verarbeitet bzw. der Abfrage an Stelle der Platzhalter hinzugefügt. Man könnte das mit einer Art "Quarantäne"-Zustand vergleichen. Diese Beschreibung ist natürlich eine grobe Vereinfachung, erleichtert aber das Verständnis für das Grundprinzip von "prepared statements".

SQL ist nicht gleich SQL - Was PDO nicht kann

Auch wenn sich mit mit PDO verschiedene Datenbank-Systeme ansprechen lassen, befreit das den Entwickler nicht davon, die Abfrage spezifisch für die Datenbank zu formulieren. Da auf der Welt verschiedene Auffassungen darüber bestehen, was ein SQL-Statement ist und was man damit alles anstellen kann, gleicht das Formulieren der Queries in einer ungewohnten Umgebung dem Gang übers Glatteis. Kein Mensch kennt alle Nuancen und Feinheiten aller SQL-Dialekte auswendig.
Ausweg für diesen Entwickler-Horror bringt eine sog. Datenbankabstraktionsschicht (Database Abstraction Layer). Vereinfacht gesagt ist das ein Versuch, eine Art "Esperanto" (Erkl. bei Wikipedia) für Datenbank-Abfragen zu schaffen. Das Projekt, das man hier erwähnen muss ist Doctrine. Es arbeitet mit der Über-Sprache "Doctrine Query Language (DQL)". Doctrine ist aber leider im Einsatz nicht ganz trivial. Ich werde nicht weiter darauf eingehen.
